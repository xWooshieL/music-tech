# 0. Главное расхождение в постановке (обязательно зафиксировать)

В вопросе сквозит формулировка «модель **генерирует** партитуру по токенам, мы просим сделать последовательность из нот». Это описание **music generation** (MusicGen, Music Transformer, RL-Duet, ReaLchords). Тезисы — про **другое**: про **score following**, отслеживание партитуры в реальном времени. Партитура **уже известна заранее**, модель её не сочиняет.

Что на самом деле делает текущая система и что описывает статья:

- **Вход системы**: (а) известная заранее партитура солиста (JSON со списком нот); (б) онлайн-поток MIDI-событий с цифрового пианино/файла; (в) известная оркестровая MIDI-дорожка.
- **Выход «модели» (трекера)**: индекс ноты в партитуре, в которой сейчас, по её оценке, находится солист. Это **не генерация нот**, а **локализация позиции в готовой партитуре**.
- **Выход системы в целом**: воспроизведение оркестрового MIDI с тем темпом, который диктует трекер.

В тезисах прямо в [10, 11] упомянуты RL-Duet и ReaLchords — это как раз генерация нового материала, и авторы тезисов специально **отгораживаются**: «(iii) RL применяется к генерации нового музыкального материала [10, 11] … Ни одна из перечисленных работ не рассматривает RL-агента как тонкий слой опережающего предсказания темпа поверх сохраняемого классического трекера».

То есть RL-агент в нашей задаче не выдаёт ноты. Он выдаёт **одно число**: темповый коэффициент `a_t ∈ [0.5, 2.0]` на горизонте 200–500 мс. Всё, что играется оркестром — это заранее записанный оркестровый MIDI, ускоренный/замедленный в `a_t` раз.

Если эта картинка совпадает с вашей — ниже подробный разбор того, что есть в репо, и план. Если расхождение принципиальное и вы хотите именно генерацию аккомпанемента нот — это другая задача (RL-Duet/ReaLchords), и переиспользовать OLTW почти не получится; сообщите, чтобы я перестроил план.

## 1.2. Формат входа в модель

Их два, потому что есть «оффлайн-партитура» и «онлайн-исполнение».

**Партитура (score)** — JSON, схема в `midi_to_score.py`:

```198:206:d:\Projects-13-03-2026\piano\midi_to_score.py
notes.append(
    {
        "index": len(notes),
        "pitches": pitches,
        "nominal_onset": round(onset_time, 6),
        "nominal_duration": round(duration, 6),
    }
)
```

То есть одна «нота» партитуры — это аккорд (`pitches: list[int]`), его номинальное время начала и номинальная длительность. Полифония свёрнута в **состояния**: одновременные нажатия (внутри `chord_epsilon=0.03 с`) кластеризуются в один state. Поэтому `pitches` — список, а не один пятитч.

**Исполнение (performance)** — поток MIDI-событий, `note_on` с `velocity > 0`. Внутри трекера каждое событие имеет вид `{"pitch": int, "timestamp": float}`. См. `dataset_viewer.load_performance`, `live_midi_receiver.LiveMidiReceiver`, `output_dispatcher.iter_midi_note_events`.

**Состояние HMM/HSMM** — скрытое состояние `i ∈ {0, ..., N-1}`, **одно состояние = одно score-state партитуры**. Эмиссионная вероятность от наблюдаемого pitch — гауссиана по полутонам:

```134:137:d:\Projects-13-03-2026\piano\hmm_follower.py
def _emission_probabilities(self, observed_pitch: float) -> np.ndarray:
    deltas = (observed_pitch - self.pitches) / self.sigma
    emission = self._gaussian_norm * np.exp(-0.5 * np.square(deltas))
    return np.maximum(emission, self._tiny)
```

Никаких «токенов» в смысле GPT/MusicGen здесь нет. Pitch — это просто число MIDI (0–127), не word-piece. И **никакой «генерации» нет**: трекер не предсказывает следующую ноту, он отвечает «вот эта пришедшая нота скорее всего соответствует состоянию №k партитуры».

## 1.3. Длительности — как соблюдаются

Здесь сделано правильнее, чем в простом HMM, и это уже почти **HSMM** в академическом смысле, хотя название класса — `ScoreFollowerHSMM`.

**В партитуре** у каждого state хранится `nominal_duration`. Forward-обновление в HSMM меняет переходные вероятности в зависимости от того, сколько времени уже проведено в текущем state:

```348:363:d:\Projects-13-03-2026\piano\hsmm_follower.py
if ratio < 1.0:
    stay_probability = 0.82
    advance_probability = 0.18
    skip_probability = 0.0
    leap_probability = 0.0
elif ratio <= 1.5:
    phase = (ratio - 1.0) / 0.5
    stay_probability = 0.35 + (0.18 - 0.35) * phase
    advance_probability = 0.60 + (0.70 - 0.60) * phase
    skip_probability = 0.03 + (0.06 - 0.03) * phase
    leap_probability = 0.02 + (0.06 - 0.02) * phase
else:
    stay_probability = 0.12
    advance_probability = 0.68
    skip_probability = 0.08
    leap_probability = 0.12
```

То есть длительность нот учитывается не через явное `p(d)`-распределение (как в Cont 2010 [5]), а через **кусочно-линейную функцию от `ratio = elapsed/nominal`**. Это эвристика, и это **первое место, где честный HSMM из тезисов даст улучшение**.

**Темп и воспроизведение оркестра** — `TempoTracker` в `output_dispatcher.py`. Он смотрит, насколько фактический промежуток между score-states отличается от номинального, и сглаживает `tempo_ratio` (медиана + deadzone). Этот ratio передаётся в `PygameMidiOrchestra` / `real_orchestra_player.py`, которые проигрывают оркестровый MIDI с соответствующей скоростью.

**Что регулируется сейчас**:
- темп: да, через `tempo_ratio`;
- громкость (velocity): **захардкожена** в `PygameMidiOrchestra.velocity = 76`, не адаптируется;
- акценты / артикуляция: нет, только то, что записано в самом оркестровом MIDI;
- педаль: только то, что в MIDI;
- баланс между голосами: нет.

То есть «звучание» сейчас определяется качеством исходного оркестрового MIDI + одним числом `tempo_ratio`. RL-агент из тезисов добавляет **опережающее** управление этим самым числом.

## Этап 2 — HMM/HSMM в академическом виде

Текущие `hmm_follower.py` и `hsmm_follower.py` — это рабочие эвристики. Чтобы превратить их в честный baseline для статьи, нужно:

1. **Явные распределения длительности** (по Cont 2010): для каждого score-state хранить параметры `p_i(d)` — например, log-normal с медианой = `nominal_duration` и обучаемым `σ_d`. Заменить кусочно-линейный код в `_transition_probabilities` на правильное HSMM-обновление `α_t(i) = Σ_τ Σ_j α_{t-τ}(j) · a_{j,i} · p_i(τ) · ∏ b_i(o_{t-τ+1:t})`. Это можно сделать с beam-урезанием, чтобы укладываться в 20 мс.
2. **Структурные переходы** (по Nakamura 2015): repeat (вернуться на k назад), insertion (лишняя нота), deletion (пропущенная нота). У нас уже есть `skip`/`leap` — но Накамура даёт их корректную интерпретацию через скрытые «error events».
3. **Beam-search forward** для N=3791. Уже сейчас numpy справляется за <1 мс, но при HSMM с явной свёрткой по `τ` это станет O(N·D), где D — размер ядра длительности. Beam k=64 даёт O(k·D), запас по latency сохраняется.
4. **Хранение и логирование α-распределения** во времени — понадобится как часть `s_t` для RL.

Файлы, которые надо тронуть/создать:
- `hsmm_follower.py` — переписать `_transition_probabilities` и `_predict_prior` под честный HSMM;
- `evaluation/` (нет) — новая папка с `evaluate_hmm.py`, `evaluate_hsmm.py`, агрегированными CSV/JSON-отчётами;
- `datasets/` (нет) — конвейер ASAP/MAESTRO.

## Этап 3 — RL-слой опережающего предсказания темпа

Реализация по тезисам (РИС. 1 + формула 1):

1. **Среда (Gymnasium API)** `score_following_env.py`:
   - `reset()`: загружает пару (score, performance), сбрасывает HSMM в начало;
   - `step(a_t)`: проигрывает чанк 200–500 мс с темпом `a_t`, обновляет HSMM по реальным событиям из performance, считает награду `r_t`;
   - `observation`: вектор `s_t = concat(α-pool, τ_{t-K:t}, e_{t-K:t}, φ̂/N)` размерности ~32–64.
2. **Сжатие α** — `α̂_t`: не передавать в политику все N значений, а взять, например, `(max(α), entropy(α), argmax(α)/N, sum of top-3 mass, top-3 indices относительно current_index)`. Это укладывается в 5–8 чисел и не зависит от длины пьесы.
3. **Behavior cloning стадия (Stage A)**: на ASAP-исполнениях подсчитать «оракульный» темп `a*_t = ΔΦ_nominal / ΔΦ_real` на скользящем окне, обучить MLP на регрессию `s_t → a*_t`. Это даёт хорошую инициализацию, без чего PPO с нуля будет сходиться плохо.
4. **PPO стадия (Stage B)** — `stable-baselines3.PPO`, награда из формулы (1), λ и μ — гиперпараметры (стартовать λ=1.0, μ=0.5; сетка). **Симулятор солиста** для онлайн-rollout-ов: брать ASAP-исполнения и проигрывать их с искажениями (jitter, rubato из модели Repp). Заменять реальную запись синтетическими отклонениями важно — иначе PPO будет переобучаться под конкретные записи.
5. **Замер latency**: всё это должно отвечать за <20 мс на CPU. Маленький MLP (2 слоя по 64) — порядка 50 мкс на инференс, безопасно.

Файлы:
- `rl/env.py`, `rl/state.py`, `rl/reward.py`, `rl/train_bc.py`, `rl/train_ppo.py`, `rl/simulator.py` (модель rubato).

## Этап 4 — рендерер и звучание

Чтобы оркестр звучал «адекватно» помимо одного `a_t`, нужно решить, что из этого исследовательская часть, а что инженерная:

- **Темп** — управляется RL-агентом, ядро тезисов.
- **Громкость (velocity)** оркестра — сейчас фиксирована. Минимальное улучшение, без новых статей: копировать сглаженный velocity солиста (`v_orch = clip(α·v_solo + (1-α)·v_score, 1, 127)`). Это **не часть тезисов**, но без этого звучание плоское.
- **Артикуляция/педаль/баланс голосов** — оставить как есть в оркестровом MIDI, не вводить второй RL-агент. Иначе scope взрывается.
- **Опережение/задержка** (out-of-thesis, но в духе): сейчас `FIRST_AUTOPLAY_DELAY = 0.14 с` (см. `autoplay_offset_benchmark.py`) — это эмпирическая компенсация задержки рендера. С RL это можно потенциально выучить, но как stretch goal.

## Этап 5 — вход от музыканта

Здесь сразу решение, не открывая новый фронт:

- **MIDI (текущее, рекомендую оставить)**: `live_midi_receiver.py` + `MidiEmulator` — оба готовы. Латентность от нажатия до события — порядка 5–10 мс, что комфортно укладывается в бюджет 20 мс.
- **Audio**: потребует онсет-детектор и pitch-tracker (Onsets-and-Frames от Magenta, или CQT-фичи как у Cont). Это **отдельная подсистема** и отдельный риск. Я бы вынес её за scope текущей работы по тезисам и оставил гипотетической надстройкой над эмиссионной моделью (`_emission_probabilities` принимала бы chroma-вектор вместо одного pitch).

# 5. Главные риски и где они порвут план

1. **ASAP даёт alignment по нотам, но не по «соло+оркестр»**. Для расчёта `t_render − t_perf` нужен «идеальный» референс — как «оркестр должен был сыграть». Решение: считать `t_perf` по solo-MIDI, а `t_render` симулировать через детерминированный рендер оркестрового MIDI с темпом `a_t` (это и есть гипотеза тезисов). То есть оркестрового аудио иметь не обязательно — мы оцениваем рассинхронизацию через таймстемпы.

2. **Latency честного HSMM** при N≈3800. Без beam — порядка 10–50 мс на свёртку по `d`, что на грани. С beam k=64 — безопасно. Это нужно проверить как первый микро-эксперимент.

3. **Award shaping в PPO**: λ и μ выбираются методом тыка. Готовиться к 2–3 проходам по гиперпараметрам.

4. **Дрейф между «офлайн-метриками» (alignment accuracy) и «онлайн-ощущением» (звучание)**: классическая проблема. Минимум — записать 5–10 коротких MIDI-исполнений руками для качественной оценки.

5. **Симулятор солиста переобучает агента**. Если параметрическая модель rubato (Repp) слишком регулярна, агент выучит её, а не общие свойства rubato. Митигация: учить BC на реальном ASAP, а PPO дообучать ограниченное количество шагов с маленьким learning-rate и KL-штрафом к BC-политике — это ровно идея [13] Sequence Tutor.

# 6. Что я предлагаю как первый практический шаг

Прежде чем писать новый HSMM с нуля, имеет смысл:

1. **Подтвердить или поправить мою постановку** (раздел 0): если вы согласны, что мы делаем score-following + RL-темп, идём дальше; если хотите именно генерацию аккомпанемента — план перестраивается.
2. **Загрузить ASAP** и написать `datasets/asap/import.py` под наш формат `score.json` + `performance.json`. Без реального датасета все следующие пункты висят в воздухе.
3. **Прогнать текущий `hybrid_fusion` на ASAP** через расширенный `evaluate.py` — получим честный baseline, от которого будут отталкиваться все улучшения.

Если согласны с такой постановкой и порядком — могу следующим шагом сделать импортер ASAP и вынести оценочные метрики из `autoplay_offset_benchmark.py` в отдельный модуль, чтобы их можно было применять и к реальным записям, и к синтетике.
